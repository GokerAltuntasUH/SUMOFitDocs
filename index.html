SUMOFit: Adjusts road networks for safer alternatives.

This documentation describes the structural components and interfaces of the SUMOFit framework.
Algorithmic details, optimisation strategies, and learning mechanisms are intentionally abstracted.
Detailed methodological and algorithmic components are reserved for forthcoming scientific publications Last updated: 2026-03-19

Modules and Functions

Domain

The domain module contains of intermediate layers of the multi-domain framework, providing a structured approach to managing the core elements of the SUMOFit. The module is designed to be the single access point for the other execution environments to interact with the core elements of the SUMOFit, including the Network class and the FlowSet class.

🌐 Network Class:
The collection of all the infrastructure-related elements.

The Network class can be taken as the Infrastructure equivalent of the FlowSet class. The class is designed to store all Network-related elements in memory at runtime, including the training phase. The module consists of three sub-modules called "managers", which are designed to provide quick access to the Network elements.

Properties

  • meta data
    A metadata class instance associated with the Network. Practically, it contains information about:
    • Export date from the Civil3D model
    • NetExport version (a complementary project) that exported the model info
    • Source of the model, defaults to "Civil3D"
    • A unique identifier of the Network element
    • HotMap code (a complementary project) that identifies the location
    • BIM file version addressing the revisions in the Existing Condition modelling
    • Whether the default flows by the SUMOFit would be applied
    • Total number of primary nodes in the network
    • Total number of transition nodes in the network
    • Total number of auxiliary nodes in the network
    • Total number of pole nodes in the network
    • Total number of edges in the network
  • node manager
    The NodeManager class instance that is associated with the Network class instance.
  • edge manager
    The EdgeManager class instance that is associated with the Network class instance.
  • chain manager
    The ChainManager class instance that is associated with the Network class instance.
🔀 Flow Set Class:
Collection of the FlowElement class instances travels through the Network

The FlowSet class can be taken as the demand equivalent of the Network class. A FlowSet class instance is generated from the network, automatically calculating all the possible flows and discovering loops with the potential to act as a Roundabout.

Properties

  • flows
    A dictionary of FlowElement class instances, indexed by the FlowId
  • default vehicle
    A default vehicle type is applied to FlowElement instances within the FlowSet.
  • flow count
    An integer corresponds to the number of FlowElement instances in the FlowSet instance.

Methods

  • from network:
    Creates the FlowSet instance from the given Network instance.

Data Managers

Data managers are designed to provide the methods needed to maintain and process data stored in memory. The practical function of the data_managers module is to help keep domain module elements, including the sub-modules, clean and organised.

🔗 Chain Manager Class
Designed to control Network-level entities involving both Nodes and Edges: EdgeChains and Loops

ChainManager is designed to handle operations that require network-level knowledge.

Properties

  • A list of edge chains
  • A list of topological loops

Methods

  • Loop Methods:
    • calculate loop center:
      Calculates and registers the loop centre for "SuperNode" state.
    • calculate loop shape:
      Calculates and registers the overall loop layout for "SuperNode" state.
  • Edge Chain Methods:
    • calculate chain shape:
      Updates the shape of the EdgeChain instance to guarantee a cross-domain alignment
➖ Edge Manager Class
An Edge is the single segment between two nodes. See also: EdgeChain

Properties

  • edges
    A list of all the Edge class instances in the network.

Methods

  • Access Methods
    • get edge:
      Gives access to a specific edge instance.
    • get edges by node:
      Returns a list of edges connecting to the same Node class instance.
    • get all edges in radial order:
      Returns a list of edges that connect to the Node class instance, ordered by their rotation.
🔹 Node Manager Class
Controls all the types of nodes, except the SuperNode, a realisation of Loops. See: Realisation Form Enumeration

Properties

  • Primary Nodes
  • Auxiliary Nodes
  • Transition Nodes
  • Pole Nodes

Fuctions of node types are explained in the Node Type Enumeration.

Methods

  • Access Methods
    • get node:
      Returns the Node class instance by the NodeGUID.
    • get primary nodes:
      Returns a list of all the PrimaryNode instances in the network.
    • get auxiliary nodes:
      Returns a list of all the PrimaryNode instances in the network.
    • get pole nodes:
      Returns a list of all the PrimaryNode instances in the network.
    • get transition nodes:
      Returns a list of all the PrimaryNode instances in the network.
  • Processors
    • calculate network bounds:
      Calculates the minimum and maximum coordinates of the Node class instances across the entire network.
    • normalise node values:
      Returns a tuple of ML-compatible Node features.
🆔 IdRegistry Class
Helps assign a unique id for elements of the Network class

Data Structures

Designed to store and organise lunch-time memory data. The structures below are designed to serve as an interface between SUMO (Alvarez Lopez et al., 2018) and Civil3D elements. The domain module elements, described above, wrap the data structures and are used by multiple execution environments.

➖ Edge Class:
A single segment between two nodes, see also EdgeChain

An Edge object helps construct the EdgeChain. Practically, any characteristic change in the Civil3D alignment within the same region will trigger a new Edge class element. Edge class elements cluster together to create an EdgeChain class instance.

Properties

  • Edge Guid
    Unique string value that represents an instance of the class
  • Edge Start Node Guid
    String representation of the origin Node class instance
  • Edge End Node Guid
    String representation of the target Node class instance
  • Edge Priority
    SUMO (Alvarez Lopez et al., 2018) priority of the Edge class instance.
🔗 Edge Chain Class:
Collection of Edges separated only by Auxiliary Nodes.

In SUMO (Alvarez Lopez et al., 2018), Edges may have a polyline geometry. An "Edge" in SUMOFit represents a single segment, while an EdgeChain represents the entire Edge in SUMO.

Properties

  • Edges
    A list of Edge class elements, always ordered by the Civil3D alignment station value that the Edge class element overlaps.
  • Location Offset
    Calculated by ChainManager class in consideration of the total road width and the Civil3D assembly insertion line.
  • shape
    Calculated by ChainManager class using the LocationOffset property of the EdgeChain class instance. The value is directly projected onto SUMO (Alvarez Lopez et al., 2018) to ensure the physical centre of the road alignment.
🔢 Enumerators:
Helps implement string-type properties by assigning integers

Helps implement string-type properties by assigning integers

Elements:

  • Realisation Form Enumeration
    The state that defines the realisation form of the abstract loop elements indentified in the road network topology.
    • Free Flow:
      Profiles are extruded along abstract lines
    • Roundabout:
      Profiles are extruded along abstract lines. Edge priorities are set to form a roundabout
    • Super Node:
      The entire loop geometry collapses into one super node

  • Node Type Enumeration
    Helps categorise the Node elements
    • Primary Node:
      Represents the intersections of abstract, linear elements, as well as the intersection of corridors
    • Transition Node:
      Represents where the abstract line features are consistent, but the corridor properties change (i.e. an extra lane)
    • Auxiliary Node:
      Represents where the corridor properties are consistent, but the abstract line features change (i.e. the beginning of a curvature).
    • Pole Node:
      Represents the endpoints of the network. Also represents the Origins and the Destinations of Flow Elements.

  • Edge Type Enumeration
    Helps categorise the Edge Elements. Edge types do not yet affect the simulation. In the future, we expect to include "spiral" type of edges in addition to "Arc"
    • Connecting:
      Restores the topographic structure in which two primary nodes are located adjacent to each other.
    • Line:
      Represents a direct connection between two nodes
    • Arc:
      Represents a curve between two nodes. Using Arc-type edges helps define curvatures without segmenting the road geometry.

  • Conflict Type Enumeration
    Replicates SUMO (Alvarez Lopez et al., 2018)-defined conflict types
    • Further information can be found at https://sumo.dlr.de/docs/Simulation/Output/SSM_Device.html
➡️ Flow Element Class:
A class that is to align with SUMO (Alvarez Lopez et al., 2018)'s internal flow mechanism

Represents a series of EdgeChains to pass through when travelling from specific origins to destinations.

Properties

  • vehicle Type Id
    Refers to the type of vehicle in the series that will follow the flow during the simulation.
  • edges
    An ordered series of Edge class instances that the vehicles in the flow will follow.
  • begin
    The simulation time at which the flow begins.
  • end
    The simulation time at which the flow exterminates.
  • vehicles per hour
    Total number of vehicles to be deployed into the simulation between the start and the end
🛣️ Lane Configuration Class:
Corresponds representation of Civil3D assemblies in SUMO (Alvarez Lopez et al., 2018) environment.

A LaneConfiguration instance contains a list of Lane definitions, each representing a sub-assembly in Civil3D. A LaneConfiguration element also handles metadata, such as the total road width.

Properties

  • Lanes
    A list of Lane definitions, each of which corresponds to another Civil3D sub-assembly type.
  • Assembly Id
    Civil3D Id of the assembly that the LaneConfiguration is rooted from.
  • RoadWidth
    Total width of the road, considering all the Lane elements.
♾️ Loop Class:
Internal loops identified in the Network class instance during the creation of the allocated FlowSet class instance.

Topologically, identified loops imply a potential to become a Roundabout as the realisation form. Elements related to the loop (Edges, Nodes, FlowElements) respond automatically.

Properties

  • nodes
    A list the nodes in the Loop
  • Loop Realisation Form
    The RealisationFormEnumeration reports whether the loop has a FreeFlow, SuperNode, or Roundabout state.
  • LoopShape
    The overall layout of the loop in the SuperNode form.

Methods

  • Access Methods
    • get next edge: Given an Edge, it returns the next Edge element in the loop according to traffic flow.
    • get previous edge: Given an Edge, it returns the previous Edge element in the loop according to the traffic flow.
    • get next node: Given a Node, it returns the next Node in the loop according to the traffic flow.
    • get previous node: Given a Node, it returns the previous Node in the loop according to the traffic flow.
🔹Node Class:
Represents points of characteristic changes in the road corridor's geometry

The Node class represents the parent definition for subclasses. The following subclasses are defined explicitly, but not mentioned in this documentation, as all the subtypes inherit from the base Node class.

  • Primary Node:
    Represents the road >3-leg road junctions.
  • Auxiliary Node:
    Controls the horizontal alignment of the EdgeChain polyline-based geometry.
  • Transition Node:
    Indicates a shift in the road assembly, without being interrupted by a PrimaryNode.
  • Pole Node:
    Indicates a dead end in the network, where the network extends in real life but is limited by the scope in SUMOFit.

Properties

  • Node Type
    An enumeration reports the node type, matching the subclass.
  • Node X
    Projection of the node's position over the Cardinal X axis.
  • Node Y
    Projection of the Node's position over the Cardinal Y axis.
🚗 Vehicle Type Class:
Each FlowElement in the network is followed by a series of a specific Vehicle Type

Vehicle types are a SUMO (Alvarez Lopez et al., 2018)-oriented concept that determines car features and the SSMs to be tracked.

Properties

Properties below are directly associated with SUMO (Alvarez Lopez et al., 2018) parameters, for which detailed information can be found at: https://sumo.dlr.de/docs/Definition_of_Vehicles%2C_Vehicle_Types%2C_and_Routes.html#available_vtype_attributes

  • id
    A unique string representation of the vehicle type. Currently, the simulation runs over a single vehicle type, defaulting to "car01".
  • accel
  • decel
  • sigma
  • minGap
  • length
  • maxSpeed
  • has_ssm_device
  • ssm_tracks
  • ssm_thresholds
  • ssm_trajectories
  • ssm_postevent_duration

RL Environment

🎮 SUMOFitEnv Class:
An application of the Gymnasium (Towers et al., 2024) environment over the SUMO (Alvarez Lopez et al., 2018) network

SUMOFitEnv class instances create Gymnasium environments (Towers et al., 2024) using the given Network class instance. The instance is responsible for modifying the network based on agentic decisions.

Properties

  • base network
    The Network class instance to be modified and evaluated by the SUMOFitEnv element.
  • base flowset
    The FlowSet class instance represents the auto-generated demand elements
  • test name
    A string value is given at each run.
  • scenario builder
    A SUMOSCenarioBuilder class instance that creates SUMO-specific (Alvarez Lopez et al., 2018) files at each step.
  • N_MAX
    The maximum number of nodes that can be present in the Network class instance, including additional nodes to be created by the agent.
  • E_MAX
    The maximum number of edges that can be present in the Network class instance, including additional edges to be created by the agent.
  • Fgeom
    Number of geometric features of nodes, currently limited to X and Y coordinates.
  • max episode steps
    Number of steps to terminate an episode, if any other threshold is not met.
  • max move
    Maximum length of the move vector that can be applied to a node by the agent.
  • observation space
    A dictionary element to store the agent's observation at each step.
  • action space
    A dictionary element defines the scope of actions the agent can take.
  • last observations
    A dictionary to track the latest state of the Network class instance.

Methods

  • Stepping Logic
    • reset:
      Reconstructs the Network and the FlowSet before an episode starts.
    • step:
      Applies the action specified by the action argument, then calculates the score.
    • move node:
      Moves the node with the given node_ind by the given move vector in the action argument.
    • rform change:
      Triggers a change in the "RealisationForm" in a Loop class, if the node with the given node_id is involved in the Loop.
  • Simulation Logic
    • create sumo scenario:
      Creates the SUMO (Alvarez Lopez et al., 2018) files for the given episode and given step, and returns the step-specific directory, a common file name shared between SUMO files, and the unique TraCI label to differentiate between simulations run in parallel.
    • calculate network score:
      Given the directories, triggers a SUMO (Alvarez Lopez et al., 2018) simulation and returns the score for the current state of the Network class instance.

The IO (Input/Output) module is designed to accommodate classes and sub-modules that read and write external files.

SUMO-specific (Alvarez Lopez et al., 2018) inputs and outputs

Designed to accommodate the data traffic between SUMOFit and SUMO (Alvarez Lopez et al., 2018)

bin locator

  • find netconverter:
    Locates the directory for the netconverter. If one is not found it opens a dialogue window to let the user locate the executable.

  • find sumo:

SSMParser Class

  • Properties
    • ssm file:
      Directory that the simulation saved the SSM results.
    • collisions:
      Returns a dictionary of collisions found in SSM results.
    • near misses:
      Returns a dictionary of near misses found in SSM results.
  • Methods
    • pars:
      Parses the SSM results of the simulation from the SSM.xml file.
    • pair key:
      Gives the conflict a unique name to match the conflict entries from the perspective of the multiple road users. Returns the identical name for the conflict.
    • parse conflicts:
      Parses information from unique conflict elements across multiple road users' perspectives, ensuring the conflict is registered only once and reflected in the results.

SUMOScenarioBuilder Class

  • Properties
    • net file:
      Stores the directory of the ".net.xml" file.
    • route file:
      Stores the directory of the ".rou.xml" file.
    • cfg:
      Stores the directory of the ".sumocfg" file.
    • netconvert path:
      Stores the directory of the ".sumocfg" file.
    • seed:
      Stores the seed value used to control SUMO's (Alvarez Lopez et al., 2018) randomness.
  • Methods
    • build for episode step:
      Creates the simulation files at the specified path, using the episode and step IDs.
    • write nodes xml:
      Writes ".nod.xml" file to the given directory.
    • write edges xml:
      Writes ".edg.xml" file to the given directory.
